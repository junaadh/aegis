## Summary

Aegis is a lightweight, self-hostable authentication and identity platform designed as the authoritative identity layer for modern distributed applications.

Aegis comprises two primary components:

- **aegisd** — a stateless HTTP server handling authentication, session management, credential storage, identity data governance, and administrative operations.
- **aegis** — a CLI for database migrations, bootstrapping, administrative operations, operational troubleshooting, and configuration management.

Aegis provides:

- Multiple authentication methods: password (Argon2id), passkeys (WebAuthn Level 2), TOTP-based MFA.
- Dual session model: HTTP-only secure cookies for browser clients; short-lived bearer tokens for services and non-browser clients.
- Distinct identity classes: registered **Users** and anonymous **Guests** (convertible to Users).
- Basic role-based access control (RBAC) with extensible permission primitives.
- Comprehensive audit logging of all identity-related events.
- A rich internal API covering auth flows, session validation, identity lookup, and full admin/dashboard data access.
- TOML-based configuration with automatic environment variable compatibility.
- A first-class SDK ecosystem: `@aegis/sdk-core`, `@aegis/sdk-server`, and `@aegis/sdk-react`.

Aegis acts as the **central identity authority** while strictly limiting what identity data leaves its boundary. This design eliminates data duplication, ensures consistency across services, and provides a foundation for compliance with PDPA and GDPR.

## Motivation

Modern distributed systems require a single, authoritative identity layer. Without one, organizations encounter recurring problems:

- **Data duplication:** User email addresses, names, and statuses are copied into every service, leading to drift and inconsistency.
- **Inconsistent user state:** A user may be active in one service but disabled in another.
- **Divergent security policies:** Password complexity, MFA requirements, and session timeouts vary across services.
- **Compliance fragility:** Correcting or deleting personal data under PDPA/GDPR becomes an error-prone, multi-service operation with no guarantee of completeness.

Third-party providers (Auth0, Clerk) offer partial solutions but introduce vendor lock-in, limited data sovereignty, and unpredictable MAU-based cost scaling.

Aegis addresses these issues by providing a self-hostable alternative that delivers:

- Full ownership and control over identity data.
- Explicit, auditable exposure of a minimal identity surface to other services.
- Consistent enforcement of authentication and authorization policies.
- A compliance-aware architecture that centralizes data subject access requests.
- Unified TOML configuration with environment variable compatibility.
- Clear consistency guarantees to prevent identity data drift.
- A first-class SDK ecosystem for rapid integration in Rust and TypeScript.

## Design Philosophy

**Lightweight by Default** — Both `aegisd` and the `aegis` CLI remain small and operationally simple. Configuration is managed via a single TOML file with optional environment variable overrides.

**Identity as a System Boundary** — Identity logic is never embedded within application services; it is always delegated to Aegis via API calls.

**Strong Defaults, Optional Complexity** — Passkeys, MFA, Redis caching, and signed tokens are supported but not required. The system starts with a single PostgreSQL instance and scales incrementally.

**Single Source of Truth** — Aegis owns all identity data. Downstream services may read a constrained subset but must never store or redefine identity attributes.

**Compliance-Aware Design** — Access to identity data is auditable, minimal, and explicit. This reduces data duplication and ensures traceability for regulatory requirements.

**Strict Boundary Enforcement** — Aegis exposes identity, not a general-purpose user profile system. Non-identity data resides in application databases keyed by the Aegis `user_id`.

**Configuration as Code** — All operational parameters live in a version-controlled TOML file. The CLI provides tooling to generate, validate, and convert configurations.

**Eventual Consistency by Design** — Downstream services may cache identity data for performance but must implement explicit invalidation. Aegis provides webhook events and clear TTL guidance.

**SDK as a First-Class Concern** — The TypeScript SDK is a published, structured package family. Application developers never hand-write HTTP clients against Aegis. The SDK mirrors the API surface in three tiers: core types, server integration, and React integration.

## Components

### aegisd — Server Daemon

Responsible for:

- Exposing public, authenticated, and internal JSON HTTP APIs.
- Orchestrating all authentication flows (password, passkey, MFA).
- Validating credentials against stored cryptographic material.
- Issuing, validating, and revoking sessions (cookie and bearer).
- Storing and governing the authoritative identity schema.
- Enforcing security policies and emitting structured audit log entries.
- Serving administrative data access endpoints for dashboard and tooling use.
- Publishing identity change events via webhooks for cache invalidation.

**Operational properties:** Stateless (all state in PostgreSQL), horizontally scalable, SIGHUP-triggered config reload for non-critical parameters.

### aegis — CLI

- `aegis config init | validate | env-export | show | diff`
- `aegis migrate up | down | status | create`
- `aegis bootstrap` — create initial admin user and internal service accounts
- Admin operations: user management, role assignment, session revocation, audit log inspection
- `aegis compliance export | audit-export | breach-report`

## Scope

**Aegis Owns:**
- Identity records (Users and Guests)
- Authentication mechanics (password hashing, passkey registration, TOTP secrets)
- Session lifecycle management
- Basic RBAC (roles and permissions assigned to users)
- Audit trail of identity events
- Identity-linked metadata (email, display name, verification status)
- Configuration management and validation
- Webhook event publication for identity changes
- Admin-facing data access endpoints for all identity entities

**Aegis Does Not Own:**
- Business domain data (orders, blog posts, inventory)
- Application-specific workflows (checkout, content publishing)
- Arbitrary user profile extensions (favorite color, marketing preferences)
- Domain-level authorization logic ("can user X edit resource Y")
- Infrastructure orchestration

---

## Explicit Non-Goals

Aegis explicitly does not:

- Store full user profiles (addresses, phone numbers, preferences beyond display hints)
- Manage business workflows or payment flows
- Act as a CMS, user data platform, or CRM
- Replace domain-level authorization logic
- Provide general-purpose key-value storage for application data
- Serve as a general-purpose event bus or message queue

Violating these non-goals — by extending Aegis tables, storing business data in `metadata`, or embedding authorization logic — is an architectural defect.

## Identity Model

### User

A registered, authenticated identity with at least one active credential. Users represent human actors or service accounts with persistent, login-capable identities.

| State                  | Description                          | Allowed Operations                        |
|------------------------|--------------------------------------|-------------------------------------------|
| `pending_verification` | Email not yet verified               | Resend verification, update email, delete |
| `active`               | Fully verified and enabled           | All authenticated operations              |
| `disabled`             | Administratively suspended           | Login denied; admin can re-enable         |
| `deleted`              | Soft-deleted per compliance          | No operations; anonymized PII             |

### Guest

An anonymous, ephemeral identity with no credentials. Guests represent unauthenticated users who may interact with the system before converting to a registered User. Guests are a separate identity class, not a transient state of a User record.

- Created via `POST /v1/auth/guest`
- Optional email association via `PATCH /v1/auth/guest/email`
- Conversion to User via `POST /v1/auth/guest/convert`
- Automatic expiration after configurable TTL (default: 7 days)

## Identity Schema vs. Business Schema (Critical Boundary)

### Identity Schema (Owned by Aegis)

| Field               | Purpose                                        | Extensible?                            |
|---------------------|------------------------------------------------|----------------------------------------|
| `id`                | Primary key, immutable UUID v7                 | No                                     |
| `email`             | Primary contact point, unique                  | No                                     |
| `display_name`      | Human-readable identifier for UI              | No                                     |
| `email_verified_at` | Timestamp of email ownership confirmation      | No                                     |
| `status`            | `active`, `disabled`, `deleted`                | No                                     |
| `roles`             | Array of role names                            | Yes (via `user_role_assignments`)       |
| `metadata`          | JSONB for non-PII extensions                   | Yes (with constraints)                 |
| Credential data     | Password hashes, passkey public keys, MFA      | No (separate tables)                   |
| Session data        | Hashed tokens and expiry metadata              | No (separate tables)                   |
| Audit logs          | Immutable record of security events            | No (separate table)                    |

### Metadata Constraints

**Allowed:** UI preferences, feature flag overrides, non-sensitive display hints, client-side analytics identifiers.

**Not Allowed:** Physical addresses, phone numbers, financial data, authentication-related data, PII beyond dedicated columns, any field required for core business logic or authorization decisions.

Enforcement: application code review + optional config allowlist (`[api.metadata_allowed_keys]`). Violations are architectural defects.

### Business Profile Schema (Owned by Application Services)

All data not directly required for authentication or identity governance must reside outside Aegis. Application services own and mutate this data exclusively, keyed by `user_id` or `guest_id`.

### Boundary Rules

1. Aegis is **read-only** for downstream services (except internal trust APIs)
2. Identity fields must not be duplicated in application databases unless treated as a cached copy
3. Application services must not extend Aegis tables or schemas
4. All identity data access flows through the Aegis API
5. Extensions use the `metadata` JSONB column (within constraints) or request new fields via RFC amendment

## Identity Data Access

### Access Rules

- Identity data readable via the internal API using service-scoped bearer tokens.
- Identity data is never writable by external services.
- All access logged to `audit_logs` with request context.
- Token scopes limit which fields are returned.

### Exposed Fields (v1)

| Field            | Notes                                      | Required Scope           |
|------------------|--------------------------------------------|--------------------------|
| `id`             | Stable identifier for foreign key use      | `identity:read:id`       |
| `email`          | Required for communication flows           | `identity:read:email`    |
| `display_name`   | UI personalization                         | `identity:read:profile`  |
| `roles`          | Local authorization decisions              | `identity:read:roles`    |
| `email_verified` | Policy enforcement flag                    | `identity:read:status`   |
| `status`         | Account state for gating logic             | `identity:read:status`   |
| `metadata`       | Non-PII extensions (opt-in)                | `identity:read:metadata` |

### Never-Exposed Fields

Password hashes, TOTP secrets, passkey public keys, raw session tokens, internal audit metadata, raw cryptographic keys.

### Consistency Model

Downstream services must treat locally stored identity fields as **cached, eventually consistent copies**.

- Cache TTL: ≤ 5 minutes for non-critical identity fields.
- Webhook events: subscribe to `user.updated`, `user.deleted`, `user.status_changed`.
- Always re-fetch for: authorization decisions, billing operations, compliance actions, security-sensitive operations.

## Authentication Model

### Cookie Sessions (Browser Clients)

- HTTP-only, Secure, SameSite=Lax cookies containing a session ID.
- Sessions stored in `sessions` table with SHA-256 hashed tokens.
- Session ID: cryptographically random 32-byte value, URL-safe base64.
- Cookie signing: HMAC-SHA256 keyed from `session.secret`.
- CSRF protection: anti-CSRF tokens required for state-changing endpoints.

### Bearer Tokens (Service and Non-Browser Clients)

- Phase 1–2: Opaque tokens validated via `/v1/internal/sessions/validate`.
- Phase 3: Signed JWT (RS256/ES256) with embedded claims for stateless validation.

### Session Security Properties

| Property              | Value              | Configurable                        |
|-----------------------|--------------------|-------------------------------------|
| Session ID entropy    | 256 bits           | No                                  |
| Cookie SameSite       | Lax                | Yes (`session.cookie.samesite`)     |
| Cookie Secure         | Always             | No                                  |
| Cookie HTTPOnly       | Always             | No                                  |
| Authenticated TTL     | 7 days default     | Yes (`session.max_age`)             |
| Guest TTL             | 7 days default     | Yes (`session.guest_max_age`)       |
| Idle timeout          | 30 minutes default | Yes (`session.idle_timeout`)        |
| Rotation on privilege | Always             | No                                  |

## Credential Model

### Passwords

- Argon2id with configurable `time_cost`, `memory_cost`, `parallelism`.
- Constant-time comparison to prevent timing attacks.
- Configurable policy: min length, complexity requirements, common password check.

### Passkeys

- WebAuthn Level 2 (FIDO2). Passwordless login and step-up authentication.
- Configurable RP ID, origins, user verification, resident key preference.

### MFA (TOTP)

- RFC 6238 compatible.
- TOTP secrets encrypted at rest with AES-256-GCM (master key from `crypto.master_key`).
- Recovery codes: one-time use, bcrypt-hashed, 10 codes by default.

## Admin & Dashboard API

The internal API is split into two tiers:

**Trust API** — machine-to-machine endpoints for session validation and identity lookup, consumed by other services in the request path.

**Admin API** — read-heavy endpoints for operational dashboards, admin backends, and CLI tooling. Includes narrow write operations for privileged administrative actions.

Both tiers sit under `/v1/internal/` and require a service bearer token with appropriate scope. These endpoints must never be exposed to the public internet. Network-level isolation (Docker network, proxy routing, CIDR allow-lists) is required and non-negotiable.

### Trust API (Service-to-Service)

| Method | Path                               | Description                               | Required Scope        |
|--------|------------------------------------|-------------------------------------------|-----------------------|
| GET    | `/v1/internal/health`              | Liveness/readiness probe                  | None                  |
| POST   | `/v1/internal/sessions/validate`   | Validate a bearer token or cookie         | `session:validate`    |
| GET    | `/v1/internal/users/:id`           | Fetch minimal identity for a user ID      | `identity:read:*`     |
| GET    | `/v1/internal/users/lookup`        | Lookup user by email (exact match)        | `identity:read:email` |

**POST /v1/internal/sessions/validate — Response**

```json
// Authenticated user
{ "valid": true, "identity_type": "user", "user_id": "01HXYZ...", "guest_id": null,
  "roles": ["member"], "expires_at": "ISO8601" }

// Guest session
{ "valid": true, "identity_type": "guest", "user_id": null, "guest_id": "01HABC...",
  "roles": [], "expires_at": "ISO8601" }

// Invalid
{ "valid": false, "reason": "expired" }
// reason: "not_found" | "expired" | "revoked"
```

## Data Model

All timestamps use `TIMESTAMPTZ` (UTC). UUID v7 for all primary keys.

### `users`

```sql
CREATE TABLE users (
    id                UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    email             TEXT NOT NULL,
    email_verified_at TIMESTAMPTZ,
    display_name      TEXT NOT NULL,
    status            TEXT NOT NULL DEFAULT 'pending_verification'
                      CHECK (status IN ('pending_verification','active','disabled','deleted')),
    metadata          JSONB DEFAULT '{}'::jsonb,
    created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at        TIMESTAMPTZ,
    anonymized_at     TIMESTAMPTZ
);

CREATE UNIQUE INDEX idx_users_email ON users (LOWER(email));
CREATE INDEX idx_users_status_active ON users (status) WHERE status = 'active';
CREATE INDEX idx_users_deleted_at ON users (deleted_at) WHERE deleted_at IS NOT NULL;
CREATE INDEX idx_users_metadata_gin ON users USING GIN (metadata);
```

## SDK Architecture

The Aegis SDK is published as a TypeScript monorepo with three packages. Each layer builds on the one below it. Application developers pick the package appropriate to their context.

```
@aegis/sdk-core        ← pure types, HTTP abstraction, no runtime assumptions
         ↑
@aegis/sdk-server      ← full API access, server-only flows, cookie/header parsing
         ↑
@aegis/sdk-react       ← provider, hooks, route protection, client components
```

The Rust SDK (`aegis-sdk-rust`) remains a separate crate in the main workspace.

None of the packages contain security-critical logic. The HTTP API is canonical. SDKs are ergonomic wrappers and can evolve independently.

---

### `@aegis/sdk-server`

**Purpose:** Full API access for server environments. Used by admin dashboards, backend services, server actions, and route handlers. This package is server-only — never imported in browser bundles.

**Contains:**
- Full API client wrapping `@aegis/sdk-core`

## Identity & Compliance (PDPA/GDPR)

### Data Subject Access & Portability

- `GET /v1/auth/me/export` — machine-readable JSON package for the requesting user.
- `aegis compliance export --user-id <uuid>` — admin-initiated compliance package.
- All export requests logged with `event_type='compliance.export.requested'`.

### Data Correction

- User-initiated: update `display_name`, request email changes via authenticated endpoints.
- Admin-initiated: `aegis user update --id <uuid> --email <new>` with mandatory audit reason.
- Downstream services re-fetch on next read; webhooks notify for critical field changes.

### Data Deletion (Right to Erasure)

1. **Soft Delete:** `users.status = 'deleted'`, `deleted_at` populated; all sessions revoked immediately.
2. **PII Anonymization:** email → `deleted-{SHA256(email)[:12]}@anonymized.local`, `display_name` → `"[Deleted User]"`, `metadata` cleared.
3. **Credential Destruction:** password hashes, TOTP secrets, passkey credentials hard-deleted.
4. **Audit Preservation:** log entries retain anonymized identifier for forensic continuity.
5. **Downstream Notification:** `user.deleted` webhook sent to registered services.

### Access Traceability

- Append-only `audit_logs` with configurable retention (default: 7 years).
- `aegis compliance audit-export --from <date> --to <date> --format csv`

## Identity vs Authorization

Aegis provides identity primitives and basic RBAC. It is not a general-purpose authorization engine.

**Aegis provides:** authentication, identity resolution, basic RBAC (coarse-grained roles with permission strings), session lifecycle tracking.

**Aegis does not provide:** resource-level authorization, ABAC, dynamic policy evaluation, business logic gating.

```
Service request flow:
1. Validate session  → POST /v1/internal/sessions/validate
2. Fetch identity    → GET /v1/internal/users/:id  (cache ≤ 5min)
3. Coarse RBAC check → roles from identity response
4. Fine-grained authz → application-specific logic against domain data
5. Allow or deny
```

This separation prevents Aegis from becoming a god-service and keeps authorization logic close to the business data it governs.

| Condition | Severity | Action |
|-----------|----------|--------|
| Auth failure rate > 100/min | Warning | Investigate brute-force |
| DB connection pool > 90% | Warning | Scale DB or app instances |
| Active session count drops > 50% in 5min | Critical | Check health, rollback |
| Audit log write errors > 0 | Critical | Compliance risk; investigate |
| Outbox dead-lettered jobs growing | Warning | Check SMTP/webhook endpoint |

### Incident Runbook (Summary)

- **DB failure:** Failover to replica; restore from backup; `aegis migrate up`.
- **Security incident:** `aegis session revoke-all`; rotate `INTERNAL_API_TOKEN` and `SESSION_SECRET`; audit `audit_logs`.
- **Outbox failure:** `POST /v1/internal/outbox/:id/retry` for dead-lettered jobs; or `aegis webhooks replay --from <timestamp>`.

## Drawbacks

- **Additional service to operate:** Requires monitoring, backup, and update procedures.
- **Network dependency:** Services depend on Aegis for auth; circuit breakers and caching required.
- **Boundary discipline:** Separating identity from business logic is a cultural requirement as much as a technical one.
- **Expanded internal attack surface:** Admin API broadens the internal surface; strict network isolation and scope enforcement are non-negotiable.
- **SDK maintenance burden:** Three TypeScript packages require semver discipline and coordinated releases with the server API.

## Decision Log

| Decision | Rationale | Alternatives Considered |
|----------|-----------|------------------------|
| **Session Rotation** | Rotate on privilege escalation (login, MFA step-up, password/email change). Not on every request — high write volume, concurrent tab complexity. `last_seen_at` throttled at ≤ 5min. | Per-request rotation. |
| **Email Verification Gate** | `ALLOW_UNVERIFIED_LOGIN = true` default. High-trust actions gated by `email_verified` at app layer. | Hard-block login for unverified users (support burden). |
| **Rate Limiting** | Reverse proxy for coarse IP limits; Aegis for identity-aware per-account/per-flow limits. Both layers, not either/or. | App-only or proxy-only. |
| **Internal Service Trust** | Per-service bearer tokens with named scopes. IP allowlists optional. mTLS in Phase 3+. | Single shared secret; mTLS first (ops burden). |
| **User Deletion** | Soft delete + PII anonymization. Preserves FK integrity. Hard removal is an explicit admin/retention workflow. | Hard delete (breaks FK integrity); no deletion (non-compliant). |
| **UUID Version** | UUID v7: time-ordered, globally unique, index-friendly. | UUID v4 (random, worse index locality). |
| **Configuration Format** | TOML-first with env var compatibility: readable, version-controllable, type-safe. | YAML; env vars only. |
| **Admin API Design** | Separate admin tier under `/v1/internal/` with `admin:read` / `admin:write` scope split. Read-heavy; writes are narrow and explicit. | Separate admin service (ops overhead); no admin API (forces direct DB access). |
| **SDK Architecture** | Three-package TypeScript monorepo. No hand-written clients. Packages are wrappers — no security-critical logic. | Single package (no runtime isolation); no SDK (integration burden). |
| **Outbox Pattern** | PostgreSQL-native. Aegis owns the table. LISTEN/NOTIFY for dispatch. Graceful degradation. | Redis queue (BYO, adds complexity); synchronous-only (poor resilience). |
| **Consistency Model** | Eventual consistency with webhook invalidation + TTL guidance. Explicit revalidation requirements for security-sensitive ops. | Strong consistency via synchronous calls (high latency). |
| **Identity vs Authorization Split** | Aegis provides identity + coarse RBAC. Application services handle resource-level authz. Prevents god-service. | Full authorization engine in Aegis. |

### Appendix C: Service Integration Contract

**Required per consuming service:**

1. Store only `user_id` / `guest_id` as foreign keys in business tables.
2. Cache identity fields with TTL ≤ 5 minutes.
3. Implement webhook handler for `user.*` events to invalidate cache immediately.
4. Verify HMAC-SHA256 webhook signatures.
5. Re-fetch identity for authorization decisions, billing operations, compliance actions.
6. Treat Aegis unavailability as authentication failure; implement circuit breaker.
7. Never store credential data (password hashes, TOTP secrets, passkey keys).

**Compliance checklist:**
- [ ] Identity cache TTL ≤ 5 minutes
- [ ] Webhook endpoint registered for `user.*` events
- [ ] HMAC signature verification implemented
- [ ] No PII stored locally beyond cached identity fields
- [ ] Audit logs include `user_id` from Aegis for all user-initiated actions

## Final Position

Aegis is a **lightweight identity platform** that centralizes authentication and identity data while enforcing a strict, auditable boundary between identity and business domains.

**With Revision 9, the platform has three distinct operational layers:**

1. **Auth flows** — all login, signup, MFA, passkey, session, and guest operations
2. **Admin & Dashboard API** — a complete queryable surface for all identity entities with narrow, explicitly scoped write operations
3. **SDK ecosystem** — `@aegis/sdk-core`, `@aegis/sdk-server`, `@aegis/sdk-react` as the canonical integration path for TypeScript consumers

These layers are designed to compose: the Admin API provides the data, `@aegis/sdk-server` wraps it for safe server-side consumption, and `@aegis/sdk-react` provides the hydration and state management needed for a React-based dashboard or admin interface. The SDK packages contain no security-critical logic and can evolve independently of the server.

**The three critical boundaries remain intact:**
1. Identity ≠ Business Data
2. Identity ≠ Authorization
3. Aegis ≠ Data Dumping Ground

Aegis provides a minimal, secure, and compliant foundation with clear integration points. The SDK-first approach ensures consuming services never drift from the canonical API contract, and the Admin API ensures operators have full visibility without direct database access.

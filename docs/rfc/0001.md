## Summary

Aegis is a lightweight, self-hostable authentication and identity platform designed as the authoritative identity layer for modern distributed applications.

Aegis comprises two primary components:

- **aegisd** — a stateless HTTP server that handles authentication, session management, credential storage, and identity data governance.
- **aegis** — a minimal CLI tool for database migrations, bootstrapping, administrative operations, operational troubleshooting, and configuration management.

Aegis provides:

- Multiple authentication methods (password with Argon2id, passkeys via WebAuthn, and TOTP-based MFA).
- Dual session management (HTTP-only secure cookies for browser clients; short-lived bearer tokens for services and non-browser clients).
- Distinct identity classes: registered **Users** and anonymous **Guests** (convertible to Users).
- Basic role-based access control (RBAC) with extensible permission primitives.
- Comprehensive audit logging of all identity-related events.
- Controlled, read-only exposure of a minimal set of identity fields to downstream application services.
- TOML-based configuration with automatic environment variable compatibility.
- Explicit consistency guarantees for cached identity data.

Aegis acts as the **central identity authority** while strictly limiting what identity data leaves its boundary. This design eliminates data duplication, ensures consistency across services, and provides a foundation for compliance with data protection regulations such as PDPA and GDPR.

## Motivation

Modern distributed systems require a single, authoritative identity layer. Without one, organizations encounter recurring problems:

- **Data duplication:** User email addresses, names, and statuses are copied into every service, leading to drift and inconsistency.
- **Inconsistent user state:** A user may be active in one service but disabled in another.
- **Divergent security policies:** Password complexity, MFA requirements, and session timeouts vary across services.
- **Compliance fragility:** Correcting or deleting personal data under regulations like PDPA becomes an error-prone, multi-service operation with no guarantee of completeness.

Third-party identity providers (IdPs) offer partial solutions but introduce new constraints:

- **Vendor lock-in:** Migrating away from a SaaS IdP is operationally expensive.
- **Limited data sovereignty:** Sensitive identity data resides outside the organization's infrastructure.
- **Unpredictable cost scaling:** Monthly active user (MAU) pricing models can become prohibitively expensive.

Aegis addresses these issues by providing a self-hostable alternative that delivers:

- **Full ownership and control** over identity data.
- **Explicit, auditable exposure** of a minimal identity surface to other services.
- **Consistent enforcement** of authentication and authorization policies across the entire system.
- **A compliance-aware architecture** that centralizes data subject access requests and reduces the surface area for personal data.
- **Unified configuration management** via TOML with backward-compatible environment variable support.
- **Clear consistency guarantees** to prevent identity data drift across services.

Aegis is intentionally scoped to provide a modern authentication baseline—including passkeys and MFA—without introducing the operational weight of a full-featured, enterprise IdP.

## Design Philosophy

Aegis is built upon the following core principles:

**Lightweight by Default**
Both `aegisd` and the `aegis` CLI remain small, understandable, and operationally simple. The codebase prioritizes clarity over feature breadth. Configuration is managed via a single TOML file with optional environment variable overrides.

**Identity as a System Boundary**
Identity is a distinct architectural boundary. Identity logic is never embedded within application services; it is always delegated to Aegis via API calls.

**Strong Defaults, Optional Complexity**
Features like passkeys, MFA, Redis caching, and signed tokens are supported but not required for initial operation. The system can start with a single PostgreSQL database and scale incrementally.

**Single Source of Truth**
Aegis owns all identity data. Downstream services may read a constrained subset of this data but must never store or redefine identity attributes.

**Compliance-Aware Design**
Access to identity data is:
- Auditable (every read and write is logged).
- Minimal (only the fields necessary for downstream operation are exposed).
- Explicit (access requires authenticated, scoped service tokens).

This design reduces data duplication and ensures traceability for regulatory requirements.

**Strict Boundary Enforcement**
Aegis exposes *identity*, not a general-purpose user profile system. All non-identity data (e.g., shipping addresses, preferences, avatars) must reside in application-specific databases, keyed by the Aegis `user_id`.

**Configuration as Code**
All operational parameters are defined in a version-controlled TOML configuration file. The CLI provides tooling to generate, validate, and convert configurations for different deployment environments.

**Eventual Consistency by Design**
Downstream services may cache identity data for performance, but must implement explicit invalidation strategies. Aegis provides webhook events and clear TTL guidance to maintain system-wide consistency without coupling.

## Components

### aegisd — Server Daemon

The Aegis server (`aegisd`) is a stateless HTTP daemon responsible for:

- Exposing the public and internal JSON HTTP APIs.
- Orchestrating all authentication flows (password, passkey, MFA).
- Validating credentials against stored cryptographic material.
- Issuing, validating, and revoking sessions (both cookie-based and bearer tokens).
- Storing and governing the authoritative identity schema.
- Enforcing security policies (e.g., rate limiting stubs, failed attempt tracking).
- Emitting structured audit log entries.
- Loading and validating configuration from TOML or environment variables.
- Publishing identity change events via webhooks for cache invalidation.

**Operational Properties:**
- **Stateless:** All persistent state resides in PostgreSQL. Multiple instances of `aegisd` can run concurrently behind a load balancer.
- **Backed by PostgreSQL:** Utilizes `pgx` with migrations managed by the CLI.
- **Horizontally Scalable:** No in-memory state is shared between instances.
- **Configuration Hot-Reload:** Supports SIGHUP-triggered configuration reload for non-critical parameters (e.g., logging levels, rate limits).

### aegis — CLI

The Aegis CLI (`aegis`) provides the operator interface for administrative and maintenance tasks:

- **Configuration Management:**
  - `aegis config init` — Generate a new `aegis.toml` with defaults.
  - `aegis config validate` — Validate configuration syntax and semantics.
  - `aegis config env-export` — Convert TOML config to `.env` format for compatibility.
  - `aegis config diff` — Compare current config against running instance.

- **Database Migrations:** 
  - `aegis migrate up` / `aegis migrate down` / `aegis migrate status`.

- **Bootstrapping:** 
  - `aegis bootstrap` to create the initial admin user and necessary internal service accounts.

- **Administrative Operations:** 
  - Manual user creation, role assignment, session revocation, and audit log inspection.

- **Operational Tooling:** 
  - Health checks, debug utilities, and diagnostic reports.

The CLI is a separate binary to keep the server runtime free of operational coupling and to minimize the attack surface of the running daemon.

## Scope

**Aegis Owns:**
- Identity records (Users and Guests).
- Authentication mechanics (password hashing, passkey registration, TOTP secrets).
- Session lifecycle management.
- Basic RBAC (roles and permissions assigned to users).
- Audit trail of identity events.
- A minimal set of identity-linked metadata (email, display name, verification status).
- Configuration management and validation.
- Webhook event publication for identity changes.

**Aegis Does Not Own:**
- Business domain data (e.g., orders, blog posts, inventory).
- Application-specific workflows (e.g., checkout, content publishing).
- Arbitrary user profile extensions (e.g., favorite color, marketing preferences).
- Infrastructure orchestration (Kubernetes manifests, Dockerfiles) — though examples are provided.
- Domain-level authorization logic (e.g., "can user X edit resource Y").

---

## Explicit Non-Goals

Aegis is deliberately scoped to avoid feature creep and maintain architectural clarity. The following are explicit non-goals:

**Aegis does NOT:**
- Store full user profiles (addresses, phone numbers, preferences beyond basic display hints).
- Manage business workflows (order processing, content publishing, payment flows).
- Act as a CMS, user data platform, or customer relationship management system.
- Replace domain-level authorization logic (fine-grained resource access decisions).
- Provide general-purpose key-value storage for application data.
- Handle billing, subscription management, or financial transactions.
- Serve as a general-purpose event bus or message queue.

These concerns remain the responsibility of application services. Aegis provides identity primitives that application services consume; it does not absorb application logic.

**Violating these non-goals**—by extending Aegis tables, storing business data in `metadata`, or embedding authorization logic—constitutes an architectural defect that undermines the platform's core value proposition.

## Identity Model

Aegis defines two distinct identity types:

### User
A registered, authenticated identity with at least one active credential (password, passkey, or MFA factor). Users represent human actors or service accounts with persistent, login-capable identities.

**User Lifecycle States:**
| State | Description | Allowed Operations |
|-------|-------------|-------------------|
| `pending_verification` | Email not yet verified | Resend verification, update email, delete |
| `active` | Fully verified and enabled | All authenticated operations |
| `disabled` | Administratively suspended | Login denied; admin can re-enable |
| `deleted` | Soft-deleted per compliance | No operations; anonymized PII |

### Guest
An anonymous, ephemeral identity with **no credentials**. Guests are used to represent unauthenticated users who may interact with the system (e.g., adding items to a cart) before converting to a registered User. Guests may optionally provide an email address to facilitate conversion.

> **Important:** Guests are a separate identity class, not a transient state of a User record. This separation simplifies session handling and data retention policies.

**Guest Lifecycle:**
- Created via `/v1/auth/guest` endpoint.
- Optional email association via `/v1/auth/guest/email`.
- Conversion to User via `/v1/auth/guest/convert` (requires credential setup).
- Automatic expiration after configurable TTL (default: 30 days of inactivity).

## Identity Schema vs. Business Schema (Critical Boundary)

Aegis enforces a strict architectural separation between **identity data** and **business profile data**.

### Identity Schema (Owned by Aegis)
This schema contains only the fields strictly required for authentication, identity resolution, access control, and compliance.

| Field               | Purpose                                       | Extensible? |
|---------------------|-----------------------------------------------|-------------|
| `id`                | Primary key, immutable UUID v7                | No          |
| `email`             | Primary contact point, unique                 | No          |
| `display_name`      | Human-readable identifier for UI               | No          |
| `email_verified_at` | Timestamp of email ownership confirmation     | No          |
| `status`            | `active`, `disabled`, `deleted`               | No          |
| `roles`             | Array of role names (e.g., `["admin"]`)       | Yes (via `user_role_assignments`) |
| `metadata`          | JSONB for future-proofing (non-PII only)      | Yes (with constraints) |
| Credential data     | Password hashes, passkey public keys, MFA secrets | No (separate tables) |
| Session data        | Hashed tokens and expiry metadata             | No (separate tables) |
| Audit logs          | Immutable record of security events            | No (separate table) |

**Characteristics:**
- Globally consistent across all services.
- Minimal and purpose-limited.
- Authoritative for all identity questions.
- Contains compliance-sensitive PII.
- `metadata` JSONB column allows non-critical, non-PII extensions without schema migrations (e.g., `{"preferred_locale": "en-US"}`).

### Metadata Constraints

The `metadata` column is strictly limited to **non-PII, non-critical identity extensions**. This guardrail prevents schema drift and boundary violations.

**Allowed in `metadata`:**
- UI preferences (e.g., `{"locale": "en-US", "theme": "dark"}`)
- Feature flag overrides (e.g., `{"beta_features": ["new_dashboard"]}`)
- Non-sensitive display hints (e.g., `{"avatar_hint": "initials"}`)
- Client-side analytics identifiers (e.g., `{"anon_session_id": "..."}`)

**Explicitly NOT allowed in `metadata`:**
- Physical addresses or location data
- Phone numbers or contact details beyond `email`
- Financial data (payment methods, billing addresses)
- Authentication-related data (backup emails, security questions)
- Any field required for core business logic or authorization decisions
- PII beyond what is already in dedicated columns

**Enforcement:**
- Application code reviews must verify `metadata` usage.
- Aegis MAY validate `metadata` keys against an allowlist (configurable via `[api.metadata_allowed_keys]`).
- Violations of these constraints are considered architectural defects and must be remediated.

### Business Profile Schema (Owned by Application Services)
All data that is not directly required for authentication or identity governance **must** reside outside Aegis.

**Examples:**
- Shipping and billing addresses.
- Phone numbers (unless used for MFA, which is handled internally).
- User preferences (theme, language, notifications).
- Profile avatars (only the reference URL may be stored; the binary asset is external).
- Domain-specific metadata (e.g., loyalty points balance, membership tier).

**Rules for Application Services:**
- All business profile tables must be keyed by the `user_id` or `guest_id` provided by Aegis.
- Application services **own and mutate** this data exclusively.
- Application services **must never write** into Aegis-managed tables.
- Application services MAY cache identity fields locally but MUST treat Aegis as authoritative and implement cache invalidation via webhooks or polling.

### Boundary Rules Summary
1.  Aegis is **read-only** for downstream services (except for internal trust APIs for session validation).
2.  Identity fields (`email`, `display_name`, etc.) **must not be duplicated** in application databases unless strictly required for indexing or referential integrity (in which case they must be treated as a cached copy, with the Aegis record remaining authoritative).
3.  Application services **must not extend** Aegis tables or schemas.
4.  All identity data access must flow through the Aegis API.
5.  Extensions to identity behavior must use the `metadata` JSONB column (within constraints) or request new fields via RFC amendment.

**Violation of this boundary leads to:**
- Data inconsistency across services.
- Increased compliance risk during data subject access or deletion requests.
- Loss of a centralized, auditable identity trail.
- Architectural debt that compounds with each violation.

## Identity Data Access

### Purpose
The identity data access layer is designed to:
1.  Eliminate duplication of identity fields across the system.
2.  Ensure downstream services consume a consistent, up-to-date view of identity.
3.  Support PDPA/GDPR compliance by centralizing access logs and minimizing data exposure.
4.  Enable fine-grained access control via service token scopes.

### Access Rules
- Identity data is readable via the internal API using service-scoped bearer tokens.
- Identity data is **never writable** by external services (updates occur only via user-authenticated flows or administrative CLI actions).
- All access to identity data is logged to the `audit_logs` table with request context.
- Token scopes limit which fields are returned (e.g., `identity:read:basic` vs `identity:read:full`).

### Exposed Fields (v1)
The following fields are exposed to trusted internal services, scoped by token permissions:

| Field               | Notes                                   | Required Scope |
|---------------------|-----------------------------------------|----------------|
| `id`                | Stable identifier for foreign key use.  | `identity:read:id` |
| `email`             | Required for communication flows.       | `identity:read:email` |
| `display_name`      | Used for UI personalization.            | `identity:read:profile` |
| `roles`             | Used for local authorization decisions. | `identity:read:roles` |
| `email_verified`    | Boolean flag for policy enforcement.    | `identity:read:status` |
| `status`            | Account state for gating logic.         | `identity:read:status` |
| `metadata`          | Non-PII extensions (opt-in exposure).   | `identity:read:metadata` |

### Non-Exposed Fields
The following fields are **never exposed** via the API under any circumstances:
- Password hashes
- MFA TOTP secrets (encrypted at rest)
- Passkey credential IDs and public keys
- Raw session tokens (only validation status is returned)
- Internal audit and security metadata
- `deleted_at` timestamps (abstracted via status field)
- Raw cryptographic keys or derivation parameters

### Field Exposure Control
Service tokens include a `fields` claim that explicitly lists permitted fields. Example token payload:
```json
{
  "sub": "service:billing",
  "scopes": ["identity:read:id", "identity:read:email", "identity:read:status"],
  "fields": ["id", "email", "status"],
  "exp": 1744819200
}
```

### Consistency Model

Aegis is the authoritative source of identity data. Downstream services must treat any locally stored identity fields as **cached, eventually consistent copies**.

To maintain consistency:

- Services MUST NOT assume cached identity data is up-to-date.
- Services SHOULD fetch fresh identity data for security-sensitive decisions.
- Services MAY cache identity fields for performance, but MUST implement one of:
  - Webhook-driven invalidation (recommended)
  - Time-based TTL expiration
  - Explicit revalidation on critical operations

### Recommended Strategy (v1)

- **Cache TTL:** ≤ 5 minutes for non-critical identity fields.
- **Webhook Events:** Subscribe to `user.updated`, `user.deleted`, `user.status_changed` to invalidate cache immediately.
- **Always Re-fetch Identity For:**
  - Authorization decisions (role checks, permission evaluation)
  - Billing or payment operations
  - Compliance-sensitive actions (data export, deletion requests)
  - Any operation where stale identity data could cause security or correctness issues

### Webhook Integration for Cache Invalidation

Aegis publishes identity change events to registered webhook endpoints:

```json
{
  "event": "user.updated",
  "timestamp": "2026-04-15T10:30:00Z",
  "actor_id": "usr_01HXYZ...",
  "data": {
    "id": "usr_01HXYZ...",
    "changed_fields": ["email", "display_name"],
    "previous_values": {
      "email": "old@example.com"
    }
  }
}
```

**Supported Event Types:**
- `user.created`, `user.email_verified`, `user.status_changed`, `user.deleted`
- `user.roles_changed` (for RBAC cache invalidation)
- `session.revoked` (for session cache cleanup)

**Webhook Delivery Guarantees:**
- At-least-once delivery with exponential backoff retry.
- HMAC-SHA256 signature verification using shared secret.
- Configurable timeout and failure threshold per endpoint.

**Service Implementation Pattern:**
```go
// Example: Billing service cache invalidation
func (s *BillingService) onUserUpdated(event UserUpdatedEvent) {
    // Invalidate cached identity data for this user
    s.identityCache.Delete(event.Data.ID)
    
    // Optionally prefetch fresh data if user is active
    if event.Data.Status == "active" {
        go s.prefetchIdentity(event.Data.ID)
    }
}
```

## Authentication Model

### Cookie Sessions (Browser Clients)
- **Mechanism:** HTTP-only, Secure, SameSite=Lax cookies containing a session ID.
- **Storage:** Sessions are stored in the `sessions` table with SHA-256 hashed tokens.
- **Lifecycle:** Managed by `aegisd` with configurable absolute (`session.max_age`) and idle (`session.idle_timeout`) timeouts.
- **Security:** 
  - Sessions are rotated upon privilege escalation (e.g., MFA verification) and on logout.
  - Session ID is a cryptographically random 32-byte value encoded as URL-safe base64.
  - Cookie signing uses HMAC-SHA256 with key derived from `session.secret`.
- **CSRF Protection:** State-changing endpoints require anti-CSRF tokens for cookie-authenticated requests.

### Bearer Tokens (Service-to-Service and Non-Browser Clients)
- **Mechanism:** Short-lived opaque tokens passed via the `Authorization: Bearer <token>` header.
- **Token Format (Phase 1-2):** Opaque string, validated via `/v1/internal/sessions/validate` introspection endpoint.
- **Token Format (Phase 3):** Signed JWT (RS256 or ES256) with embedded claims for stateless validation.
- **Validation:** 
  - Opaque tokens: Database lookup with cache (Redis Phase 2).
  - JWTs: Signature verification + claim validation (exp, nbf, scopes).
- **Use Cases:** API clients, mobile applications, and internal service mesh communication.
- **Token Refresh:** Optional refresh token flow for long-lived client sessions (opt-in via config).

### Session Security Properties
| Property | Value | Configurable |
|----------|-------|-------------|
| Session ID entropy | 256 bits | No |
| Cookie SameSite | Lax | Yes (`session.cookie.samesite`) |
| Cookie Secure | Always (HTTPS required) | No |
| Cookie HTTPOnly | Always | No |
| Absolute timeout | 7 days default | Yes (`session.max_age`) |
| Idle timeout | 30 minutes default | Yes (`session.idle_timeout`) |
| Rotation on privilege change | Always | No |

## Credential Model

### Passwords
- **Hashing:** Argon2id with parameters tuned for the target environment.
- **Default Parameters:** 
  ```toml
  [credentials.password.argon2]
  time_cost = 3
  memory_cost = 65536  # 64 MB
  parallelism = 2
  salt_length = 16
  hash_length = 32
  ```
- **Validation:** Constant-time comparison to prevent timing attacks.
- **Breached Password Detection:** (Future) Integration with HIBP API or local k-anonymity check; configurable via `[credentials.password.hibp]`.
- **Password Policy:** Enforced at registration/update; configurable rules:
  ```toml
  [credentials.password.policy]
  min_length = 12
  require_uppercase = true
  require_lowercase = true
  require_digit = true
  require_symbol = true
  disallow_common = true  # Check against top 10k common passwords
  disallow_email_substring = true
  ```

### Passkeys (v1)
- **Standard:** WebAuthn Level 2 (FIDO2).
- **Flows Supported:**
    - Passwordless login (primary authentication).
    - Step-up authentication (MFA context).
- **Attestation:** None required by default; configurable via `[credentials.passkeys.attestation]`.
- **Relying Party Configuration:**
  ```toml
  [credentials.passkeys]
  rp_id = "auth.example.com"
  rp_name = "Example Corp"
  origins = ["https://app.example.com", "https://admin.example.com"]
  user_verification = "preferred"  # "required", "preferred", "discouraged"
  resident_key = "preferred"
  timeout_seconds = 60
  ```
- **Credential Storage:** Public keys stored in `passkey_credentials` table; credential IDs are user-provided authenticator identifiers.

### Multi-Factor Authentication (MFA) (v1)
- **TOTP:** RFC 6238 compatible.
  - Secrets encrypted at rest using AES-256-GCM with key derived from the database encryption key (`crypto.master_key`).
  - Configurable parameters:
    ```toml
    [credentials.mfa.totp]
    issuer = "Example Corp"
    period = 30
    digits = 6
    algorithm = "SHA1"
    skew = 1  # Allow ±1 time step for clock drift
    ```
- **Passkey as Second Factor:** Passkeys can be used for step-up authentication.
- **Recovery Codes:** 
  - One-time use, bcrypt-hashed recovery codes generated during MFA enrollment.
  - Default: 10 codes, each 12 characters (6 pairs of uppercase letters/digits).
  - Configurable via `[credentials.mfa.recovery_codes]`.
- **MFA Policy Enforcement:** 
  - Per-role or per-user MFA requirements configurable via RBAC.
  - Step-up MFA for sensitive operations (configurable endpoint list).

### Credential Lifecycle
- **Rotation:** Passwords and TOTP secrets can be rotated via authenticated endpoints.
- **Revocation:** Individual credentials (e.g., a specific passkey) can be revoked without affecting others.
- **Backup:** Recovery codes serve as backup authentication; stored hashed, never displayed again after generation.

## Configuration Management

Aegis adopts a **TOML-first configuration model** with backward-compatible environment variable support. This approach provides:
- Human-readable, version-controllable configuration files.
- Type-safe validation at startup.
- Hierarchical overrides (file → environment → CLI flags).
- Automatic generation of `.env` files for deployment tooling.

### Configuration File Format (`aegis.toml`)

```toml
# aegis.toml — Aegis Configuration
# Version: 1.0
# Schema: https://aegis.example.com/schema/config/v1

[server]
# Network binding
host = "0.0.0.0"
port = 8080
public_url = "https://auth.example.com"

# TLS (optional; termination often handled by reverse proxy)
tls_cert = "/secrets/cert.pem"
tls_key = "/secrets/key.pem"

# Logging
log_level = "info"  # debug, info, warn, error
log_format = "json"  # json, text
log_output = ["stdout"]  # stdout, stderr, /path/to/file

[database]
# PostgreSQL connection (required)
url = "postgresql://aegis:***@db.example.com:5432/aegis"
max_connections = 25
min_idle = 5
connection_timeout_seconds = 30

# Encryption at rest for sensitive columns
encryption_key = "env:DB_ENCRYPTION_KEY"  # Reference env var or use "file:/path/to/key"

[redis]
# Optional: caching, rate limiting, distributed locks (Phase 2+)
enabled = false
url = "redis://cache.example.com:6379/0"
max_connections = 10

[session]
# Cookie session configuration
secret = "env:SESSION_SECRET"  # 32-byte base64 key
max_age_hours = 168  # 7 days
idle_timeout_minutes = 30
cookie:
  name = "aegis_session"
  path = "/"
  domain = "example.com"
  secure = true
  http_only = true
  same_site = "lax"  # strict, lax, none

# Bearer token configuration
bearer:
  opaque_token_length = 32  # bytes
  default_ttl_minutes = 15
  refresh_token_enabled = false
  refresh_token_ttl_days = 30

[credentials]
# Password settings
[credentials.password]
hashing_algorithm = "argon2id"
[credentials.password.argon2]
time_cost = 3
memory_cost = 65536
parallelism = 2
[credentials.password.policy]
min_length = 12
require_uppercase = true
require_lowercase = true
require_digit = true
require_symbol = true
disallow_common = true

# Passkey/WebAuthn settings
[credentials.passkeys]
rp_id = "auth.example.com"
rp_name = "Example Corp"
origins = ["https://app.example.com"]
user_verification = "preferred"
resident_key = "preferred"
timeout_seconds = 60

# MFA/TOTP settings
[credentials.mfa]
[credentials.mfa.totp]
issuer = "Example Corp"
period = 30
digits = 6
algorithm = "SHA1"
skew = 1
[credentials.mfa.recovery_codes]
count = 10
code_length = 12

[email]

  host = "smtp.example.com"
  port = 587
  username = "env:SMTP_USERNAME"
  password = "env:SMTP_PASSWORD"
  starttls = true
  timeout_seconds = 30

# Email templates (optional custom paths)
templates:
  verification = "/etc/aegis/templates/verify.html"
  password_reset = "/etc/aegis/templates/reset.html"

[api]
# Rate limiting (basic; advanced via reverse proxy)
rate_limit:
  enabled = true
  requests_per_minute = 60
  burst = 10

# Internal API trust
```

### Environment Variable Compatibility

Aegis supports automatic conversion between TOML and `.env` formats:

**Conversion Rules:**
1.  Nested TOML keys become prefixed environment variables: `[section.subsection] key = "value"` → `SECTION_SUBSECTION_KEY=value`
2.  Special references: `env:VAR_NAME` in TOML directly uses the environment variable.
3.  File references: `file:/path/to/secret` reads the file content at startup.
4.  Boolean, integer, and duration types are parsed automatically.

**CLI Commands for Configuration:**

```bash
# Generate a new config with interactive prompts
aegis config init --output aegis.toml

# Validate configuration syntax and semantics
aegis config validate --config aegis.toml

# Convert TOML to .env format for deployment tools
aegis config env-export --config aegis.toml --output .env

# Show effective configuration (after env overrides)
aegis config show --config aegis.toml --format json

# Diff running config against file
aegis config diff --config aegis.toml --endpoint http://localhost:8080
```

**Environment Variable Override Precedence:**
1.  CLI flags (highest priority)
2.  Environment variables (`AEGIS_SERVER_PORT=9000`)
3.  TOML configuration file
4.  Built-in defaults (lowest priority)

**Example Environment Variable Mapping:**
| TOML Path | Environment Variable | Example Value |
|-----------|---------------------|---------------|
| `server.port` | `AEGIS_SERVER_PORT` | `8080` |
| `database.url` | `AEGIS_DATABASE_URL` | `postgresql://...` |
| `session.secret` | `AEGIS_SESSION_SECRET` | `base64key...` |
| `credentials.password.policy.min_length` | `AEGIS_CREDENTIALS_PASSWORD_POLICY_MIN_LENGTH` | `12` |

### Configuration Validation

At startup, `aegisd` performs comprehensive validation:
- Schema validation against embedded JSON Schema.
- Cross-field constraints (e.g., `tls_cert` requires `tls_key`).
- Secret presence checks for required sensitive values.
- Network reachability tests for database/SMTP/Redis (optional, `--validate-connections` flag).

Invalid configurations cause immediate exit with descriptive error messages.

## Data Model (Extended)

The database schema is normalized, extensible, and designed for auditability. All timestamps use `TIMESTAMPTZ` (UTC). UUID v7 is used for time-ordered, index-friendly primary keys.

### Core Identity Tables

#### `users`
```sql
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    email TEXT NOT NULL UNIQUE,
    email_verified_at TIMESTAMPTZ,
    display_name TEXT NOT NULL,
    status TEXT NOT NULL CHECK (status IN ('pending_verification', 'active', 'disabled', 'deleted')),
    metadata JSONB DEFAULT '{}'::jsonb,  -- Non-PII extensions only (see constraints)
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at TIMESTAMPTZ,
    
    -- Indexes
    CONSTRAINT email_unique_case_insensitive UNIQUE (LOWER(email))
);

CREATE INDEX idx_users_status_active ON users (status) WHERE status = 'active';
CREATE INDEX idx_users_deleted_at ON users (deleted_at) WHERE deleted_at IS NOT NULL;
CREATE INDEX idx_users_metadata_gin ON users USING GIN (metadata);
```

**Extensibility Notes:**
- `metadata` JSONB allows non-critical, non-PII fields without schema changes.
- PII must never be stored in `metadata`; use dedicated columns or external systems.
- GIN index enables efficient JSONB queries for admin tooling.

#### `guests`
```sql
CREATE TABLE guests (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    email TEXT,  -- Optional, unique if provided
    metadata JSONB DEFAULT '{}'::jsonb,  -- Same constraints as users.metadata
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    converted_to UUID REFERENCES users(id) ON DELETE SET NULL,
    expires_at TIMESTAMPTZ NOT NULL,
    
    CONSTRAINT email_unique_if_present UNIQUE (email)
);

CREATE INDEX idx_guests_expires_at ON guests (expires_at);
CREATE INDEX idx_guests_converted ON guests (converted_to) WHERE converted_to IS NOT NULL;
```

**Lifecycle Management:**
- Expired guests are purged via scheduled job (`aegis cron cleanup-guests`).
- Conversion to user preserves guest `metadata` via merge logic.

#### `sessions`
```sql
CREATE TABLE sessions (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    token_hash BYTEA NOT NULL UNIQUE,  -- SHA-256 of session token
    user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    guest_id UUID REFERENCES guests(id) ON DELETE CASCADE,
    expires_at TIMESTAMPTZ NOT NULL,
    last_seen_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    mfa_verified BOOLEAN NOT NULL DEFAULT false,
    user_agent TEXT,
    ip_address INET,
    metadata JSONB DEFAULT '{}'::jsonb,  -- Client context, non-sensitive
    
    -- Ensure only one of user_id or guest_id is set
    CONSTRAINT session_identity_check CHECK (
        (user_id IS NOT NULL AND guest_id IS NULL) OR
        (user_id IS NULL AND guest_id IS NOT NULL)
    )
);

CREATE INDEX idx_sessions_token_hash ON sessions (token_hash);
CREATE INDEX idx_sessions_user_active ON sessions (user_id, expires_at) 
    WHERE expires_at > NOW() AND user_id IS NOT NULL;
CREATE INDEX idx_sessions_guest_active ON sessions (guest_id, expires_at) 
    WHERE expires_at > NOW() AND guest_id IS NOT NULL;
CREATE INDEX idx_sessions_cleanup ON sessions (expires_at) WHERE expires_at < NOW();
```

**Security Properties:**
- `token_hash` prevents token leakage from database compromise.
- `mfa_verified` enables step-up authentication tracking.
- Partitioning by `expires_at` recommended for high-volume deployments (Phase 2+).

#### `password_credentials`

```sql
    attestation_object BYTEA,     -- Optional attestation data
    authenticator_data BYTEA NOT NULL,
    sign_count BIGINT NOT NULL DEFAULT 0,
    transports TEXT[],            -- ["usb", "nfc", "ble", "internal"]
    backup_eligible BOOLEAN NOT NULL DEFAULT false,
```

#### `totp_credentials`
```sql
CREATE TABLE totp_credentials (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v7(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    secret_encrypted BYTEA NOT NULL,  -- AES-256-GCM ciphertext
    nonce BYTEA NOT NULL,             -- GCM nonce
    algorithm TEXT NOT NULL DEFAULT 'SHA1',
    digits INT NOT NULL DEFAULT 6,
    period INT NOT NULL DEFAULT 30,
    backup_codes_encrypted BYTEA[],   -- Array of bcrypt-hashed codes (encrypted)
    enabled BOOLEAN NOT NULL DEFAULT true,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    UNIQUE (user_id)  -- One TOTP factor per user (Phase 1)
);

CREATE INDEX idx_totp_credentials_user ON totp_credentials (user_id);
```

**Encryption Details:**
- `secret_encrypted` = AES-256-GCM(plaintext_secret, key=derived_from_master_key, nonce=random)
- Master key derivation: HKDF-SHA256(master_key, salt='totp', info='aegis-v1')

```sql
    used_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

);
```

## Identity & Compliance (PDPA/GDPR)

Aegis is designed to simplify compliance with data protection regulations by centralizing identity data operations and providing auditable, minimal data exposure.

### Data Subject Access & Portability (Article 15 / Section 26)
- **Single Endpoint:** `GET /v1/auth/me/export` (authenticated) returns a machine-readable JSON package containing all identity data for the requesting user.
- **Admin Tooling:** `aegis compliance export --user-id <uuid>` generates a compliance package with audit trail.
- **Format:** JSON with embedded schema documentation; optional CSV export for specific fields.
- **Audit:** All export requests logged to `audit_logs` with `event_type='compliance.export.requested'`.

### Data Correction (Right to Rectification)
- **User-Initiated:** Users can update `display_name` and request email changes via authenticated endpoints.
- **Admin-Initiated:** `aegis user update --id <uuid> --email <new>` with mandatory audit reason.
- **Propagation:** Downstream services fetch updated identity data on next read; optional webhook notification for critical fields.
- **Audit:** All corrections logged with before/after values (PII redacted in logs).

### Data Deletion (Right to Erasure / Right to be Forgotten)
Aegis implements a **soft delete + anonymization** strategy to balance compliance with referential integrity:

1.  **Soft Delete:** 
    - `users.status` set to `deleted`, `deleted_at` populated.
    - All active sessions revoked immediately.
    - User can no longer authenticate.

2.  **PII Anonymization:**
    - `email` overwritten with deterministic placeholder: `deleted-{SHA256(email)[:12]}@anonymized.local`
    - `display_name` set to `"[Deleted User]"`.
    - `metadata` cleared of any PII (validated via schema).

3.  **Credential Destruction:**
    - Password hashes, TOTP secrets, passkey credentials permanently deleted (hard delete).
    - Recovery codes purged.

4.  **Audit Preservation:**
    - Audit log entries referencing the user retain the anonymized identifier.
    - `actor_id` in logs preserved for forensic continuity.

5.  **Downstream Notification:**
    - Optional webhook `user.deleted` sent to registered services.
    - Services responsible for anonymizing their business profile data keyed by `user_id`.

**CLI Commands:**
```bash
# Request deletion (user self-service)
POST /v1/auth/me/delete  # Requires password confirmation

# Admin-initiated deletion with reason
aegis user delete --id <uuid> --reason "GDPR Article 17 request" --notify-downstream

# Verify deletion state
aegis user show --id <uuid>  # Returns status=deleted, anonymized fields
```

### Access Traceability & Audit
- **Comprehensive Logging:** Every authentication attempt, data access, and administrative action logged to `audit_logs`.
- **Immutable Records:** Audit logs are append-only; updates require new entries with correction metadata.
- **Retention Policy:** Configurable via `[compliance.data_retention.audit_log_days]`; default 7 years.
- **Export for Auditors:** `aegis compliance audit-export --from <date> --to <date> --format csv` generates redacted reports.

### Data Processing Agreements (DPA) Support
- **Data Residency:** Configuration option `[database.region]` documents physical data location.
- **Subprocessor Disclosure:** Aegis documentation lists all third-party dependencies (e.g., SMTP provider).
- **Breach Notification:** `aegis compliance breach-report --incident-id <id>` generates template notifications.

## Identity vs Authorization

Aegis provides identity primitives and basic role-based access control (RBAC). It is explicitly **not** a general-purpose authorization engine.

### What Aegis Provides (Identity Layer)
- **Authentication:** Verifying "who you are" via credentials.
- **Identity Resolution:** Mapping sessions/tokens to user/guest records.
- **Basic RBAC:** Assigning coarse-grained roles (`admin`, `user`, `service`) with permission strings.
- **Session Management:** Tracking authenticated state and privilege levels.

### What Aegis Does NOT Provide (Authorization Layer)
- **Resource-level authorization:** Decisions like "can user X edit document Y" must be handled by application services.
- **Attribute-based access control (ABAC):** Policy evaluation based on resource attributes, context, or relationships.
- **Dynamic permission evaluation:** Runtime policy engines (e.g., OPA, Cedar) remain the responsibility of application services.
- **Business logic gating:** Workflow permissions (e.g., "only authors can publish their own drafts") are application concerns.

### Integration Pattern
```
Application Service Request Flow:
1. Receive request with user session/token
2. Validate session with Aegis: POST /v1/internal/sessions/validate
3. Fetch identity: GET /v1/internal/users/{id} (cached with invalidation)
4. Evaluate coarse-grained role: if "admin" in user.roles { ... }
5. Evaluate fine-grained authorization: application-specific logic
   - Query domain database for resource ownership
   - Apply business rules, ABAC policies, or workflow state
6. Allow or deny request based on combined identity + authorization decision
```

### Role Permission Strings (Aegis RBAC)
Permissions are coarse-grained signals, not exhaustive policy:
- `identity:read:*` — Read any identity field (scoped by token)
- `admin:user:*` — Administrative user management
- `service:billing:read` — Service-specific permission for billing context

Application services interpret these signals within their domain context. Aegis does not evaluate whether a permission applies to a specific resource.

**This separation prevents Aegis from becoming a god-service** and keeps authorization logic close to the business data it governs.

## Deployment Model

Aegis is designed for containerized deployment (Docker, Kubernetes) with infrastructure-as-code support.

- **PostgreSQL** 15+ (required, persistent storage).

- **Redis** 7+ (optional, Phase 2: caching and distributed rate limiting).
- **SMTP Server** (required for email verification and recovery flows).
- **TLS Termination:** Recommended at reverse proxy (nginx, Traefik) or via `aegisd` native TLS.

## Scaling Path

Aegis is designed to scale horizontally as demand grows, with clear phase gates.

| Phase | Description | Key Changes | Target Scale |
|-------|-------------|-------------|--------------|
| **1** | **Database-backed sessions.** Single `aegisd` instance with PostgreSQL. | - All sessions stored in `sessions` table.<br>- Rate limiting at reverse proxy.<br>- Opaque bearer tokens with introspection.<br>- Webhook events for cache invalidation. | ≤ 10k MAU, ≤ 100 RPS |
| **2** | **Redis integration.** Add Redis for session caching, rate limiting, and OTP storage. | - Session cache with LRU eviction.<br>- Token bucket rate limiting in-app.<br>- TOTP code cache to prevent replay.<br>- `api_access_logs` table for observability. | ≤ 100k MAU, ≤ 1k RPS |
| **3** | **Signed tokens (JWT) & stateless validation.** Introduce JWT support for internal service authentication. | - JWT issuance with RS256/ES256.<br>- Stateless token validation (no introspection).<br>- Token refresh flow for long-lived clients.<br>- Session secret rotation support. | ≤ 1M MAU, ≤ 10k RPS |
| **4** | **Advanced features & federation.** OAuth/OIDC, multi-tenancy, fine-grained RBAC. | - `user_external_identities` table.<br>- Organization scoping.<br>- Permission policy engine.<br>- Plugin system for custom MFA providers. | Enterprise scale |

### Scaling Considerations
- **Database:** 
  - Read replicas for identity lookup endpoints (`/v1/internal/users/:id`).
  - Connection pooling via PgBouncer for high concurrency.
  - Table partitioning for `audit_logs` and `sessions` by time.
- **Application Layer:**
  - Stateless `aegisd` instances behind load balancer.
  - Graceful shutdown with session drain (Phase 2+).
- **Caching Strategy:**
  - L1: In-memory LRU cache for hot identity records (per-instance).
  - L2: Redis distributed cache for session tokens and rate limit counters.

### Performance Targets (Phase 1)
| Operation | P50 Latency | P99 Latency | Throughput |
|-----------|-------------|-------------|------------|
| Password login | ≤ 300ms | ≤ 800ms | 200 RPS/instance |
| Session validation | ≤ 50ms | ≤ 150ms | 2000 RPS/instance |
| User lookup (internal) | ≤ 30ms | ≤ 100ms | 5000 RPS/instance |
| Audit log write | ≤ 20ms | ≤ 80ms | 1000 RPS/instance |
| Webhook delivery (async) | ≤ 100ms | ≤ 300ms | 500 events/sec |

*Targets assume PostgreSQL on dedicated hardware, SSD storage, and adequate connection pooling.*

## Operational Considerations

### Monitoring & Observability
- **Metrics Endpoint:** `/metrics` (Prometheus format) exposing:
  - `aegis_auth_attempts_total{status="success|failure"}`
  - `aegis_session_active_count`
  - `aegis_database_connections_{used,max}`
  - `aegis_http_request_duration_seconds{endpoint,method}`
  - `aegis_webhook_delivery_attempts_total{status="success|failure"}`
- **Distributed Tracing:** OpenTelemetry support; propagate `traceparent` headers.
- **Log Aggregation:** Structured JSON logs with `request_id` for correlation.

### Alerting Recommendations
| Condition | Severity | Suggested Action |
|-----------|----------|-----------------|
| `aegis_auth_attempts_total{status="failure"}` rate > 100/min | Warning | Investigate brute-force attempt |
| Database connection pool > 90% utilized | Warning | Scale DB or app instances |
| `aegis_session_active_count` drop > 50% in 5min | Critical | Check service health, rollback if needed |
| Audit log write errors > 0 | Critical | Compliance risk; immediate investigation |
| Webhook delivery failure rate > 10% | Warning | Check endpoint health, retry configuration |

### Maintenance Procedures
- **Database Migrations:** 
  - Run `aegis migrate up` during low-traffic window.
  - Backward-compatible migrations allow zero-downtime deploys.
- **Configuration Updates:**
  - Validate with `aegis config validate` before deployment.
  - Use SIGHUP for non-breaking config changes (logging, rate limits).
- **Secret Rotation:**
  - Session secret: Requires graceful session migration (Phase 3) or forced re-login.
  - Database encryption key: Requires re-encryption job (CLI tool provided).

### Disaster Recovery Runbook (Summary)
1.  **Database Failure:** Failover to replica; restore from backup if needed.
2.  **Configuration Corruption:** Roll back to previous config version; validate before reapplying.
3.  **Security Incident:** 
    - Revoke all sessions via `aegis session revoke-all`.
    - Rotate `INTERNAL_API_TOKEN` and `SESSION_SECRET`.
    - Audit `audit_logs` for suspicious activity.
4.  **Webhook Endpoint Failure:** 
    - Temporarily disable failing endpoint via config.
    - Replay missed events using `aegis webhooks replay --from <timestamp>`.

## Drawbacks

- **Additional Service to Operate:** Introduces another moving part in the infrastructure stack, requiring monitoring, backup, and update procedures.
- **Network Dependency:** Application services become dependent on Aegis availability for authentication and identity resolution; circuit breakers and caching recommended.
- **Strict Boundary Discipline:** Requires developers to consistently separate identity concerns from business logic, which may initially conflict with rapid prototyping habits.
- **Configuration Complexity:** TOML configuration with hierarchical overrides requires team education; validation at startup mitigates runtime errors.
- **Migration Overhead:** Existing applications must refactor to delegate identity operations to Aegis; incremental adoption strategy recommended.
- **Cache Invalidation Discipline:** Services must implement webhook handling or TTL strategies to avoid identity data drift; this is a cultural as well as technical requirement.

## Decision Log

| Decision | Rationale | Alternatives Considered |
|----------|-----------|------------------------|
| **Session Rotation** | Rotate on privilege escalation (login, MFA step-up) and logout. Do not rotate on every request to avoid performance penalties and client complexity. | Rotate on every request (higher security, higher latency); fixed session lifetime only. |
| **Email Verification** | Account creation allowed without immediate verification. Downstream services gate features based on `email_verified` flag. Balances UX with security. | Require verification before any authenticated action (higher friction). |
| **Rate Limiting Strategy** | Primary rate limiting at reverse proxy/load balancer. Aegis implements fine-grained, security-focused limits (failed logins per account). | In-app rate limiting only (harder to tune); no rate limiting (insecure). |
| **Internal Service Trust** | Static bearer tokens with configurable scopes for Phase 1-2, with mandatory rotation and secret manager storage. Signed JWTs introduced in Phase 3 for granular, stateless trust. | mTLS only (complex PKI); API keys without scopes (less granular); no rotation policy (insecure). |
| **User Deletion Policy** | Soft delete with anonymization of PII to maintain foreign key integrity in downstream databases. Balances compliance with referential integrity. | Hard delete (breaks foreign keys); no deletion (non-compliant). |
| **UUID Version** | UUID v7 for all primary keys: time-ordered, globally unique, index-friendly in PostgreSQL. | UUID v4 (random, worse index locality); BIGSERIAL (not globally unique). |
| **Configuration Format** | TOML-first with env var compatibility: human-readable, version-controllable, type-safe validation. | YAML (more complex parsing); JSON (less human-friendly); env vars only (hard to manage hierarchies). |
| **Credential Storage** | Separate tables per credential type with encryption at rest for secrets. Enables independent lifecycle management. | Single `credentials` polymorphic table (complex queries); no encryption (non-compliant). |
| **Audit Log Design** | Append-only, structured JSONB metadata, partitioning-ready. Supports compliance queries without PII exposure. | Unstructured text logs (hard to query); real-time streaming only (no historical analysis). |
| **Metadata JSONB Constraints** | Explicit allowlist/denylist for `metadata` to prevent boundary violations. Enforced via code review and optional config validation. | Unrestricted JSONB (leads to schema drift); no metadata column (reduces extensibility). |
| **Consistency Model** | Eventual consistency with webhook invalidation + TTL guidance. Prevents identity drift while allowing performance optimizations. | Strong consistency via synchronous calls (high latency); no guidance (leads to drift). |
| **Identity vs Authorization Split** | Aegis provides identity + coarse RBAC; application services handle resource-level authorization. Prevents god-service anti-pattern. | Full authorization engine in Aegis (complex, coupled); no RBAC (forces duplication). |

## Future Work

The following features are explicitly out of scope for the initial release but are planned for future iterations:

### Phase 2 (6-12 months)
- **Redis Integration:** Caching layer for sessions, rate limiting, and OTP validation.
- **Webhook System Enhancements:** Dead-letter queue, replay tooling, and delivery analytics.
- **API Access Logging:** Optional `api_access_logs` table for service usage analytics.
- **Configuration Hot-Reload:** SIGHUP support for non-breaking config changes.

### Phase 3 (12-18 months)
- **External Identity Providers:** OAuth 2.0 / OIDC federation (e.g., "Sign in with Google/Apple").
- **Signed JWT Tokens:** Stateless service authentication with RS256/ES256.
- **Token Refresh Flow:** Long-lived client sessions with secure refresh mechanism.
- **Session Secret Rotation:** Graceful migration strategy for credential key rotation.

### Phase 4 (18-24 months)
- **Advanced RBAC:** Fine-grained permissions with policy engine (Rego or CEL).
- **Organization (Multi-Tenancy) Support:** Scoping users and guests to teams or organizations.
- **OIDC Bridge:** Expose Aegis as a standards-compliant OpenID Connect Provider.
- **mTLS Support:** Mutual TLS for internal service communication with automatic cert rotation.

### Phase 5+ (24+ months)
- **Async Email:** Queue-based email delivery (NATS, Kafka) for improved response times.
- **Breached Password Detection:** Integration with HIBP API or local bloom filters.
- **Plugin System:** Extension points for custom MFA providers, credential validators.
- **Geofencing & Risk-Based Auth:** Location-based access rules and anomaly detection.

### Explicit Non-Goals (Reiterated)
- Full user profile management (avatars, preferences) — remains in application services.
- Billing or subscription management — delegate to dedicated services.
- Content management or business workflow orchestration.
- General-purpose authorization engine — application services retain this responsibility.

### Appendix E: Service Integration Contract Template

**Purpose:** Define how application services integrate with Aegis while maintaining architectural boundaries.

**Required Service Responsibilities:**
1.  **Identity Fetch Pattern:**
    ```go
    // Always fetch fresh identity for security-sensitive operations
    func (s *Service) authorizeUser(userID uuid.UUID, resource Resource) error {
        // Step 1: Validate session with Aegis
        session, err := s.aegisClient.ValidateSession(ctx, token)
        if err != nil { return err }
        
        // Step 2: Fetch identity (with caching + invalidation)
        identity, err := s.identityCache.GetOrFetch(userID, func() (*Identity, error) {
            return s.aegisClient.GetUser(ctx, userID)
        })
        if err != nil { return err }
        
        // Step 3: Coarse-grained role check (Aegis RBAC)
        if !identity.HasRole("admin") && resource.IsAdminOnly() {
            return ErrForbidden
        }
        
        // Step 4: Fine-grained authorization (application logic)
        if !s.domainAuthz.CanAccess(userID, resource) {
            return ErrForbidden
        }
        
        return nil
    }
    ```

2.  **Cache Invalidation Implementation:**
    ```go
    // Webhook handler for identity changes
    func (s *Service) handleUserUpdated(event UserUpdatedEvent) {
        // Invalidate cached identity immediately
        s.identityCache.Delete(event.Data.ID)
        
        // Optional: prefetch if user is active and likely to be used soon
        if event.Data.Status == "active" {
            go func() {
                identity, _ := s.aegisClient.GetUser(context.Background(), event.Data.ID)
                if identity != nil {
                    s.identityCache.Set(event.Data.ID, identity, 5*time.Minute)
                }
            }()
        }
    }
    ```

3.  **Local Storage Rules:**
    - ✅ Store `user_id` as foreign key in all business tables.
    - ✅ Cache `email`, `display_name`, `roles` with TTL ≤ 5 minutes.
    - ❌ Never store `password_hash`, `totp_secret`, or other credential data.
    - ❌ Never duplicate `email_verified_at` without invalidation logic.

4.  **Error Handling:**
    - Treat Aegis unavailability as authentication failure (deny access).
    - Implement circuit breaker for Aegis calls to avoid cascading failures.
    - Log identity fetch failures with `request_id` for correlation.

**Compliance Checklist for Services:**
- [ ] Identity cache TTL configured ≤ 5 minutes
- [ ] Webhook endpoint registered for `user.*` events
- [ ] HMAC signature verification implemented for webhooks
- [ ] No PII stored locally beyond cached identity fields
- [ ] Audit logs include `user_id` from Aegis for all user actions

## Final Position

Aegis is more than an authentication service.

It is a **lightweight identity platform** that centralizes authentication and identity data while enforcing a strict, auditable boundary between identity and business domains. This architecture enables consistent system design, reduces operational complexity for downstream services, and provides a solid foundation for building compliance-aware applications.

By adopting a TOML-first configuration model with environment variable compatibility, Aegis balances developer ergonomics with operational rigor. The extensible data model—featuring JSONB metadata columns with explicit constraints, partitioning-ready indexes, and encryption-at-rest patterns—ensures the platform can evolve without disruptive migrations.

**Critical Boundaries Protected:**
1.  **Identity ≠ Business Data:** Aegis owns only authentication-critical fields; all profile data resides in application services.
2.  **Identity ≠ Authorization:** Aegis provides coarse RBAC signals; resource-level decisions remain with application logic.
3.  **Aegis ≠ Data Dumping Ground:** Metadata constraints and code review discipline prevent boundary erosion.

**Consistency Guaranteed:**
- Webhook-driven cache invalidation + TTL guidance prevents identity drift.
- Explicit revalidation requirements for security-sensitive operations.
- Audit trails for all identity access and changes.

Aegis does not seek to replace every identity feature in existence. Instead, it provides a minimal, secure, and compliant foundation upon which organizations can build their unique identity requirements. The strict boundary enforcement between identity and business data is not a limitation but a deliberate architectural choice that pays dividends in maintainability, auditability, and regulatory compliance.

For teams building distributed systems in regulated environments, Aegis offers a path to identity management that is both pragmatic and principled. By protecting the three critical boundaries and implementing the consistency model, Aegis remains a focused, production-grade identity layer—not a god-service.

**Next Step for Adoption:** Implement the Service Integration Contract (Appendix E) in one pilot service. Measure cache hit rates, webhook delivery latency, and authorization decision latency. Iterate on TTL values and invalidation patterns before org-wide rollout.
